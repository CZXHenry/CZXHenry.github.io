前一篇文章记录搭官方Demo的经过，但是很多Envoy的特性其实还是很模糊的，所以现在就按着自己的想法来捣鼓一下Envoy

1. 系统配置

系统配置跟前一篇文章所说的配置基本一样

  • 系统： CentOS7，虚拟机，CPU 2+ CPU核心 2+RAM 4G+VMWare
  • Docker： 17.03.2-ce
  • Docker Compose： 1.23.1
  • Git： 1.8.3.1
  • Python： 2.7.5(宿主机)，2.7.12（镜像）因为这里所用的python程序只是一个很简单的程序，没有涉及到任何python版本特性，所以版本上有些差异应该没事

2.2 往集群中增加一个服务

接下来想试试在容器中多添加一个服务进程看看会不会有负载均衡

2.2.1 Demo结构

Demo结构大部分跟原来是一样的，只不过是在容器中多加了一个服务程序.

2.2.2 配置文件

配置文件跟2.1.2中差不多，需要新增一个svc2.py程序、修改envoy-config.yaml文件和run.sh

  • svc2.py 直接复制svc1内容，不过需要制定程序监听的端口不为5000，否则在同一个容器中两个程序会产生端口冲突

  • envoy-config.yaml 在cluster.hosts中添加一个新的主机

  • run.sh 添加命令，当容器构建完成之后运行svc2

2.1.3 运行

  • 执行docker-compose up –build -d构建新的镜像
  • 执行多次 curl localhost:8100,发现主机名和主机地址完全是一样的

  • 想了一下发现是自己犯蠢了，因为svc1和svc2两个程序都是运行在同一个容器中，主机名和ip地址当然是一样的，修改了一下svc1和svc2的输出语句，然后重新构建镜像。发现是可以负载均衡是正常工作的

2.3 服务发现

服务发现说是Envoy的一个重要功能，我对这个东西的理解就是可以自动发现集群中的主机。但是就以 2.2 中的Demo来说，如果不指定host的ip地址和监听端口，envoy怎么知道应该把请求发送到对应的主机中？接下来就来看看这个服务发现究竟是个怎么回事。还有对Envoy文档中说的DNS这个东西还不是很清楚到底是个什么概念。

2.3.1 Demo结构

这个结构跟 2.2.1 中一样

2.3.2 配置文件

配置文件跟 2.2.2 中的配置文件一样，但是删除envoy-config.yaml中cluster.hosts中的第二部主机，并且将cluster.type改为strict-dns

  • envoy-config.yaml

2.3.3 运行

  • 构建镜像并运行容器
  • 执行curl localhost:8100 返回的只有service1

  • 输出容器日志，可以看出svc1跟svc2都是在运行中的，进入容器中也能访问到svc2

  • 所以这样看来貌似服务发现并不是我想的那样，突然就卡住了。然后翻了不少资料来看，看完之后貌似有点开窍了。这里的服务发现大致上是跟我想的一样，就是可以动态的管理集群中的host，而关键的地方就在于EDS Server。这个EDS Server其实可以理解为一个服务注册中心，envoy向这个服务注册中心查询当前某个集群中有多少个主机。而服务注册中心的储存的数据是动态更新的，所以envoy可以实现动态的集群host更新，而不用增加或者删除一个host都要改写一次envoy配置文件中的设置。
  • 但是这里还有两个个问题，就是服务是怎么向服务注册中心进行注册的，服务注册中心怎么监测服务的健康程度的。这个就是我们自己可以设计的了，跟envoy没有多大关系，envoy只需要知道访问这个服务注册中心就可以知道host的地址跟端口就ok。详细的可以看下下面链接的微服务:从设计到部署这篇文章

    Attention

    • 这里的EDS Server程序(即是服务注册中心程序)是要我们自己实现的
    • 相关的参考链接，可以先看一下
      • 微服务:从设计到部署-Chap4. 服务发现 比较详细的说了为什么需要服务发现、服务发现的实现类型以及可以怎么去实现服务注册
      • Envoy Discovery “hello world” 这篇就是一篇实践记录文章，作者搭建了一个简单的服务发现Demo，并且在GitHub里面提供了服务注册中心的实现，也写了他是怎么实现的服务注册（比较粗暴）。
      • Envoy 快速入门 直接从 “通过EDS动态配置cluster hosts”这一段开始看就行，中文资料，看起终究是比英文舒服点，就是看这里就突然开窍了
    • 至于DNS究竟是什么还是没搞清楚，一开始以为DNS其实就是xDS Server，但是测试过之后发现并不是

2.4 更进一步的服务发现

刚刚在2.3里面也写了大概服务发现是个怎么回事，应该怎么用，现在就来搭一个服务发现的Demo。

Attention 找了好久终于发现Envoy官方文档里面有关服务注册中心的叙述了。我这里用了服务注册中心这个说法，对应的其实就是文档里面说的 XDS Management Server。至于gRPC在这里是一个什么作用，暂时还没搞清楚。

2.4.1 Demo结构

这个Demo可能复杂一点，需要额外实现一个服务注册中心，并且需要解决服务注册问题。对于服务注册，这里应该是用简单的自注册方式来实现（其实跟用Curl这种粗暴的方法也没啥区别hhh）。这里直接返回固定的endpoint信息，而不搞什么数据库之类的东西了

2.4.2 配置文件

  • Dockerfile

  • run.sh

  • eds-server.py
    这是一个很简单的EDS server实现，没有涉及到数据库之类的东西，当收到envoy的查询请求之后就直接返回两个服务的地址和端口

  • envoy-eds-config.yaml
    这个配置文件大部分都是不变的，变化的地方主要是cluster标签下

  • docker-compose.yaml
    改动的东西不大，就是换一下envoy的配置文件即可

2.4.3 运行

  • 执行docker-compose up –build -d

  • 执行docker-compose logs service1，查看service1容器的日志

    • 可以看到一开始三个服务（svc1、svc2和eds-server）都已经正常启动
    • envoy向设定好的EDS server发送REST请求

    • eds-server返回的JSON文件被enovy转换成proto3文件并读取endpoint配置
    • envoy成功为service1添加两个endpoint
  • 执行多次curl localhost:8100 发现两个endpoint都已经被添加到了集群之中，并且有负载均衡

总结 envoy的使用暂时就做到这一步，再加上官方教程里面的前端代理搭建等，基本可以把envoy的使用摸个大致清楚了，更深入的使用可以自己进一步摸索

3. 问题及解决办法

3.1 apt-get update出现hash sum mismatch错误

  • 感觉没有万能的解决方法，只能各种试一试了。我这里后面是用了换源的方法，换了中科大的源就可以了，基本没出过什么问题

3.2 运行envoy镜像时出现错误 Unable to parse JSON as proto (INVALID_ARGUMENT:admin: Cannot find field.)

  • envoy的配置文件里面有错误，比如打错单词这些
  • 后面出了一个错误是 Unable to parse JSON as proto (INVALID_ARGUMENT:: Cannot find field.) 连是哪个标签出错都没提示，最后发现是缩进没有对齐，就是同级别标签的缩进量必须一样，这个具体解释可能得看yaml的语法了。（比如我这个就是因为http_filters跟route_config缩进没有对齐）

3.3 访问集群中的节点时显示no healthy endpoints

  • 请求被路由到了一个没有endpoint的集群，这个问题就因人而异了，我这里是cluster中hosts的地址写错了
